# 迭代修正法

## 核心原则

Plan 不是一次性输出。每轮用户反馈后重新评估：假设是否仍然成立？路径是否需要修正？

## 流程

1. 用户反馈 → 记录修正原因（写入 PlanState.modifiedReason）
2. 重新评估假设 → 哪些假设被用户确认/推翻？
3. 修正路径 → 基于新假设调整 actionSteps
4. 版本 +1 → plan-v1.md → plan-v2.md
5. 展示新旧对比 → 用户确认或再调整

## 规则

- 用户一句话否定了某假设 → 整个 Plan 必须重新评估
- 用户部分确认 → 只修正受影响的部分
- 版本历史永久保留，旧版本不删除
