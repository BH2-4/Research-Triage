# 0. 文档阅读说明

本文档是“人人都能做科研”的 MVP 正式版 PRD。

本版本的重点不是完整定义一个大型科研平台，而是明确一个最小可验证产品：

> 让一个非专业用户通过简单对话，被系统逐步识别、追问、引导，并最终获得一份适合自身基础的科研探索 Plan 和可读文档结果。

本文档遵循以下阅读顺序：

1. 先理解产品为什么存在；
2. 再明确第一版服务谁；
3. 再确定 MVP 只做什么、不做什么；
4. 然后定义用户流程、功能、页面和 AI 行为；
5. 最后用验收标准和风险控制保证可落地。

# 1. 产品概述

## 1.1 产品一句话定义

**人人都能做科研**是一款面向非专业科研人群的 AI 科研启蒙与路径引导平台，通过傻瓜式对话、多轮追问、用户画像识别、Plan 展示和文档预览，帮助用户从“我对科研感兴趣但不知道怎么开始”进入“我知道自己是谁、适合做什么、下一步怎么做”的可执行状态。

## 1.2 产品定位

本产品不是专业科研论文工具，也不是通用 AI 聊天机器人，而是一个面向普通人的科研入口。

它的定位是：

> 用普通人熟悉的交互方式，承载科研式的问题拆解、路径规划和持续引导。

用户不需要一开始就理解研究问题、论文、实验、变量、综述、数据库等概念。系统会通过对话逐步识别用户状态，并用适合用户水平的语言进行引导。

## 1.3 核心价值

产品核心价值包括四点：

1. **降低科研入口门槛**  
   让非专业用户也能开始理解和尝试科研。

2. **把模糊兴趣转化为可执行路径**  
   用户不需要一开始就提出标准科研问题，系统会帮助其拆解。

3. **根据用户画像调整解释方式**  
   不同年龄、教育背景、工具能力和认知起点的用户，应得到不同表达方式和行动建议。

4. **用 Plan 替代泛泛回答**  
   产品输出不应停留在普通聊天回答，而应生成结构化、可审查、可调整的计划。

## 1.4 MVP 版本目标

MVP 版本只验证一个核心闭环：

```text
用户输入模糊想法
↓
系统识别用户画像
↓
系统多轮追问与博弈
↓
系统生成科研探索 Plan
↓
用户审查、拆解、调整 Plan
↓
系统输出可读文档或行动清单
```

MVP 成功的标志不是功能数量多，而是：

> 一个科研新手能在不懂术语、不懂工具、不懂流程的情况下，通过对话获得一条具体、可理解、可执行的科研入门路径。

# 2. 背景与问题

## 2.1 背景：科研不应只属于专业圈层

传统科研训练通常默认用户已经在专业系统内：

- 学校；
- 实验室；
- 公司研发部门；
- 高校导师体系；
- 学术论文体系；
- 专业科研工具链。

这导致科研被很多普通人视为高门槛、远距离、精英化的事情。

但从能力本质上看，科研并不只是发表论文，而是一种更普遍的能力：

- 提出问题；
- 搜集信息；
- 验证假设；
- 形成解释；
- 记录过程；
- 分享结果。

这些能力并不应该只属于专业机构。

## 2.2 问题：普通人缺少进入科研的入口

非专业用户接触科研时，常见问题包括：

1. 不知道什么问题可以研究；
2. 不知道自己当前懂多少；
3. 不知道该从资料、实验、观察还是讨论开始；
4. 不知道如何使用 AI 和互联网工具；
5. 不知道如何把兴趣整理成行动计划；
6. 不知道如何判断自己的想法是否可行。

因此，核心问题不是用户没有兴趣，而是：

> 用户没有一个足够低门槛、足够会引导、足够贴近个人状态的科研入口。

## 2.3 产品机会：用熟悉交互承载科研引导

普通用户已经熟悉一类操作方式：

- 打开 App；
- 输入一句话；
- 点击按钮；
- 看图片；
- 看文档；
- 根据提示继续下一步。

本产品的机会在于：

> 外层使用用户熟悉的傻瓜式交互，内层运行科研分诊、画像识别、问题拆解、Plan 生成和路径引导。

用户感受到的是简单对话，系统内部完成的是科研启蒙流程。

## 2.4 当前版本的核心判断

MVP 不直接服务所有人，而是先服务一个最容易验证的核心用户：

> Z 世代普通用户 / 科研新手 / 对 AI 和科技感兴趣但没有系统科研训练的人。

原因是：

1. 该用户对 AI 和对话式产品接受度高；
2. 其工具基础足够支撑 Web 产品验证；
3. 其科研基础不足，痛点明显；
4. 其语言风格和使用习惯容易建模；
5. 可作为后续扩展到其他人群的中间基准。

# 3. 产品理念与设计原则

## 3.1 科研平权

产品的长期理念是：

> 科研不应是少数专业人士才能接触的东西，每个人都应该有理解、尝试和分享科学探索的机会。

这里的“人人都能做科研”不是指人人都能立刻完成高水平学术研究，而是指人人都应拥有进入科研思维的入口。

## 3.2 极客精神

产品体现四层极客精神：

1. **Geek for fun**  
   科研可以从兴趣和好奇心开始。

2. **Geek for sharing**  
   科研应包含分享、开源和共同进步。

3. **Geek for future**  
   普通人理解科学与 AI，是面向未来的基础能力。

4. **Geek for everyone**  
   科研入口应向更多人开放。

## 3.3 傻瓜式交互，科研式内核

产品外层应尽量简单：

- 像聊天；
- 像选择；
- 像查看文档；
- 像跟一个会追问的人讨论。

产品内层则要严谨：

- 判断用户是谁；
- 判断用户当前阶段；
- 判断用户表达中的模糊点；
- 拆解问题；
- 给出 Plan；
- 让用户确认；
- 持续修正路径。

## 3.4 先引导，再输出

普通 AI 容易在用户问题还不清楚时直接输出长答案。

本产品应避免这种模式。

核心原则是：

> 用户目标不清楚时，先追问；用户画像不清楚时，先识别；用户能力不清楚时，不直接给复杂工具；路径不清楚时，先生成可讨论的 Plan。

## 3.5 先跑通单一画像，再扩展泛化

长期愿景是全年龄段用户，但 MVP 不能一开始就泛化。

第一阶段只做：

> Z 世代科研新手画像。

后续再扩展到：

- 小学生；
- 本科生；
- 中老年用户；
- 只会手机基础操作的人；
- 历史时代模拟画像。

# 4. 目标用户与使用场景

## 4.1 长期目标用户

| 用户类型 | 用户特征 | 核心需求 |
|---|---|---|
| 小学生 / 中学生 | 好奇心强，知识体系弱 | 通俗解释、简单实验、兴趣引导 |
| Z 世代普通用户 | 熟悉互联网，但科研训练不足 | 用熟悉语言理解科研路径 |
| 本科生 / 科研新手 | 有学习能力，但不会拆课题 | 选题、综述、实验和行动路径 |
| 中老年用户 | 可能只会基础手机操作 | 低门槛、强引导、少术语解释 |
| 非技术背景用户 | 对 AI 和科技感兴趣，但不懂工具 | 工具翻译、步骤拆解、陪伴式引导 |

## 4.2 MVP 核心用户

MVP 优先服务：

> Z 世代普通用户 / 科研新手 / 对 AI 和科技有兴趣但没有系统科研训练的人。

该用户的典型特征：

- 会使用手机、网页和常见 App；
- 知道 AI 很重要，但不会系统使用 AI 做探索；
- 对科技、发明、社会问题、AI 应用等内容感兴趣；
- 不知道如何把兴趣变成研究问题；
- 需要系统用更贴近其语言习惯的方式引导。

## 4.3 用户画像识别维度

系统需要识别以下维度：

1. 年龄段 / 时代背景；
2. 教育水平；
3. 工具使用能力；
4. AI 熟悉程度；
5. 科研理解程度；
6. 兴趣方向；
7. 当前卡点；
8. 可投入设备；
9. 可投入时间；
10. 偏好的解释风格。

## 4.4 典型使用场景

### 场景 1：用户有兴趣，但不知道怎么开始

用户输入：

> 我对 AI 很感兴趣，但我什么都不懂，能不能做点科研？

系统应识别用户基础，并给出低门槛路径。

### 场景 2：用户有一个想法，但不知道是不是科研

用户输入：

> 我想研究为什么有些人总是沉迷短视频，这算科研吗？

系统应帮助用户把生活观察转化为研究问题。

### 场景 3：用户有任务，但不会拆解

用户输入：

> 老师让我做一个研究性学习，但我不知道选什么题。

系统应提供选题分诊和可执行方案。

### 场景 4：用户只会基础工具

用户输入：

> 我只会用手机和微信，可以做吗？

系统应根据工具能力给出低门槛操作路径。

## 4.5 后续画像扩展方向

MVP 跑通后，可扩展为多画像验证：

- 小学生画像；
- 中学生画像；
- 本科生科研新手画像；
- 中老年手机用户画像；
- 2021 年后开始接触 AI 的用户画像；
- 改革开放初期成长用户画像；
- 1930 年以前历史模拟画像；
- 清朝人等历史时代模拟画像。

历史时代画像不作为 MVP 真实用户，而作为测试系统理解不同认知起点的验证集。

# 5. 用户痛点与需求

## 5.1 不知道如何进入科研

用户常见状态：

- 有兴趣，但不知道从哪里开始；
- 有问题，但不知道怎么研究；
- 有任务，但不知道如何拆解；
- 有资料，但不知道如何判断有无价值。

对应需求：

> 系统需要把用户的模糊兴趣转化为明确问题、学习路径和可执行行动。

## 5.2 不知道如何使用基础工具

用户可能只会：

- 使用微信；
- 使用短视频 App；
- 使用手机拍照；
- 搜索简单关键词；
- 问 AI 一句话。

但不会：

- 设计问题；
- 追问 AI；
- 整理文档；
- 构建计划；
- 查资料并判断可信度。

对应需求：

> 产品应把复杂科研工具能力隐藏在后端，用简单前端交互承接。

## 5.3 不知道如何表达真实需求

用户往往无法准确说出：

- 我是谁；
- 我懂多少；
- 我想做什么；
- 我卡在哪里；
- 我希望 AI 帮我到什么程度。

对应需求：

> 系统不能只等待用户提出标准问题，而要主动追问、判断和修正。

## 5.4 不知道如何利用 AI 深入探索

用户常见 AI 使用问题：

- 只会问一句话；
- 不会限制输出；
- 不会要求拆解；
- 不会验证结果；
- 不会把 AI 输出变成文档或计划。

对应需求：

> 系统需要替用户完成一部分提示词工程和路径设计，让用户自然进入更高质量的 AI 对话。

## 5.5 需要清晰、可读、可执行的结果

用户不需要看到大量 Markdown 符号、复杂术语或后端推理痕迹。

用户需要：

- 看得懂；
- 知道下一步；
- 能选择继续深入或简化；
- 能获得一份像文档一样可读的结果。

对应需求：

> 产品必须具备 Plan 展示和文档预览能力。

# 6. MVP 范围

## 6.1 MVP 核心闭环

MVP 只验证一个闭环：

```text
傻瓜式输入
↓
用户画像识别
↓
多轮博弈追问
↓
Plan 生成
↓
Plan 展示与拆解
↓
文档预览
↓
人工审核迭代
```

## 6.2 P0：必须实现

### P0-1：对话入口

- 用户进入后直接看到对话框；
- 支持自然语言输入；
- 不要求用户填写复杂表单。

### P0-2：用户画像识别

- 识别用户背景、工具能力、兴趣方向和当前卡点；
- 给出初步判断；
- 允许用户纠正。

### P0-3：多轮博弈引导

- 系统必须主动追问；
- 系统必须指出模糊点；
- 系统必须在关键节点让用户选择。

### P0-4：Plan 生成

Plan 至少包括：

1. 用户当前状态；
2. 当前问题判断；
3. 推荐路径；
4. 行动步骤；
5. 风险提示；
6. 下一步选择。

### P0-5：Plan 展示

- Plan 应有独立展示区域；
- 不只是聊天气泡中的长文本；
- 支持继续拆解、简化、加深。

### P0-6：文档预览

- 展示可读文档；
- 不直接显示 Markdown 源码符号；
- 支持行动清单、研究路径、总结文档展示。

### P0-7：基础多端适配

- Web 优先；
- 手机、电脑、平板可用；
- 竖屏下内容不能严重拥挤。

## 6.3 P1：应该实现

1. 多级选择组件；
2. “拆分选项”功能；
3. 更简单 / 更专业两种解释模式；
4. 基础 Memory；
5. Prompt / Skill 模板切换；
6. 图片和文档基础展示；
7. 用户画像修正入口。

## 6.4 P2：可后置

1. Multi-Agent 完整工作流；
2. MCP 深度接入；
3. Python 自动化校验；
4. 科研数据库接入；
5. 文档深度编辑；
6. 用户空间完整文件系统；
7. 多历史时代画像；
8. 硬件入口，例如树莓派加屏幕。

## 6.5 MVP 明确不做

MVP 阶段不做：

1. 不做完整科研论文写作；
2. 不做真实科研数据库系统；
3. 不做复杂实验设计平台；
4. 不做导师端或机构端后台；
5. 不做全年龄段同时泛化；
6. 不做复杂硬件形态；
7. 不做情绪安抚模块；
8. 不做完全自动化 Multi-Agent 系统；
9. 不把历史时代画像作为主流程。

# 7. 核心用户流程

## 7.1 首次进入流程

```text
用户打开产品
↓
看到对话框和极简引导语
↓
输入一句兴趣、问题或任务
↓
系统不直接输出完整方案，而是进行初步识别
↓
系统提出 1-3 个关键问题
↓
用户回答
↓
系统进入画像判断和 Plan 准备阶段
```

## 7.2 用户画像识别流程

```text
用户输入
↓
系统识别背景信息
↓
系统识别工具能力
↓
系统识别科研理解程度
↓
系统识别当前卡点
↓
系统生成用户画像摘要
↓
用户确认或纠正
```

用户画像摘要示例：

```text
你目前更像是：Z 世代科研新手。
你对 AI 和科技有兴趣，但还没有形成科研问题。
你能使用基础网页和 App，但不熟悉科研工具。
当前最适合的路径不是直接读论文，而是先把兴趣转化为一个可验证的小问题。
```

## 7.3 多轮博弈流程

系统每轮对话遵循：

```text
拆解 → 理解 → 反问 / 对抗 → 引导 → 输出
```

含义：

1. **拆解**：拆出用户话语里的兴趣、目标、模糊点；
2. **理解**：判断用户真正想解决什么；
3. **反问 / 对抗**：指出不清楚、不现实或过大的地方；
4. **引导**：给出可选方向；
5. **输出**：形成当前阶段的小结或 Plan。

## 7.4 Plan 生成与调整流程

```text
画像确认
↓
问题确认
↓
系统生成初版 Plan
↓
前端独立展示 Plan
↓
用户选择：拆解 / 简化 / 加深 / 修改方向 / 输出文档
↓
系统调整 Plan
↓
进入下一轮确认
```

## 7.5 文档预览与输出流程

```text
Plan 稳定
↓
系统生成文档结果
↓
文档区展示富文本内容
↓
用户查看、修改、继续追问
↓
系统输出最终行动清单或研究路径
```

## 7.6 人工审核与迭代流程

MVP 阶段需要人工审核，审核重点包括：

1. 用户画像是否准确；
2. 追问是否必要；
3. Plan 是否具体；
4. 行动步骤是否可执行；
5. 语言风格是否适合用户；
6. 文档展示是否清楚。

人工审核结果用于迭代：

- Prompt 模板；
- 画像规则；
- Plan 模板；
- 追问频率；
- 前端展示方式。

# 8. 功能需求

## 8.1 功能模块总览

| 模块 | 优先级 | 功能说明 |
|---|---|---|
| 对话入口 | P0 | 用户通过聊天方式开始使用 |
| 用户画像识别 | P0 | 判断用户背景、基础、工具能力和当前卡点 |
| 多轮博弈引导 | P0 | 主动追问、拆解、对抗和修正理解 |
| Plan 生成与展示 | P0 | 输出结构化计划并展示判断逻辑 |
| 文档预览面板 | P0 | 展示可读文档、行动清单和版本内容 |
| 多端适配 | P0 | 手机、电脑、平板基础可用 |
| 多级选择 | P1 | 用户不懂时提供选项和拆分选项 |
| 图片 / 文档展示 | P1 | 支持基础材料展示 |
| Memory 机制 | P1 | 记录用户画像、偏好和进度 |
| Skill / Prompt 动态引导 | P1 | 根据画像切换引导策略 |
| Multi-Agent 工作流 | P2 | 后续扩展为多角色协作系统 |

## 8.2 对话入口

### 功能说明

用户进入产品后，应直接看到一个简洁、专业、现代感强的对话界面。

### 用户操作

用户可直接输入：

- 我想研究 AI，但我什么都不懂；
- 我想做一个科学小项目；
- 我只有手机，可以做科研吗；
- 我不知道自己的想法算不算研究。

### 系统反馈

系统不应直接输出长方案，而应先给出初步判断和关键追问。

### 验收标准

- 用户能通过一句话开始；
- 系统能进入背景识别流程；
- 首轮回复包含至少一个关键追问；
- 不输出泛泛科普长文。

## 8.3 用户画像识别

### 功能说明

系统通过对话识别用户画像，并形成内部状态。

### 识别维度

1. 年龄段 / 时代背景；
2. 教育水平；
3. 工具使用能力；
4. AI 熟悉程度；
5. 科研理解程度；
6. 兴趣方向；
7. 当前卡点；
8. 可投入时间和设备。

### 验收标准

- 至少识别 4 个关键维度；
- 能说明判断依据；
- 用户可以纠正画像；
- 纠正后系统能调整输出。

## 8.4 多轮博弈引导

### 功能说明

系统通过持续追问、拆解和判断，逼近用户真实需求。

### 行为要求

- 不过早下结论；
- 不直接给复杂方案；
- 对模糊点进行追问；
- 给用户提供选项；
- 支持“更简单 / 更专业 / 拆开讲”。

### 验收标准

- 用户目标模糊时必须触发追问；
- 系统能指出至少一个关键不确定点；
- 系统能提供 2-4 个下一步选择；
- 用户可选择拆分选项。

## 8.5 Plan 生成与展示

### 功能说明

Plan 是产品最核心的输出形态。

### Plan 内容结构

每个 Plan 至少包含：

1. 用户当前画像；
2. 当前问题判断；
3. 系统判断逻辑；
4. 推荐路径；
5. 可执行步骤；
6. 风险提示；
7. 下一步选择。

### 前端展示要求

- Plan 有独立区域；
- 支持折叠和展开；
- 支持点击某一项继续追问；
- 支持更简单、更专业、拆分、换方向。

### 验收标准

- 系统能生成结构化 Plan；
- Plan 至少包含 5 个明确部分；
- 用户能继续拆解任一部分；
- 手机竖屏下可读。

## 8.6 文档预览面板

### 功能说明

系统需要在对话之外展示文档内容，类似 ChatGPT Canvas / 文件预览。

### 展示要求

- 展示富文本；
- 不直接显示 Markdown 源码符号；
- 支持标题、列表、重点内容；
- 支持展示修改项和删减项；
- 可展示 Plan、研究路线、总结文档、行动清单。

### 验收标准

- 文档内容正常显示；
- 不出现乱码；
- 标题、列表、引用、重点内容格式正确；
- 用户能区分原文、修改项和新增内容。

## 8.7 图片与文档展示

### 功能说明

支持基础图片和文档展示。

### MVP 要求

- 图片能正常显示；
- 文档能正常预览；
- 不要求完整 Office 编辑；
- 不要求多人协同编辑。

### 验收标准

- 上传图片后可正常查看；
- 文档内容可正常预览；
- 移动端可读；
- 不出现明显格式错乱。

## 8.8 Memory 与用户状态

### 功能说明

系统需要保存用户画像、偏好和当前进度。

### 抽象结构

```text
User Space
├── profile.memory      用户画像
├── progress.memory     当前进度
├── preference.memory   交互偏好
├── plan.history        历史计划
└── prompt.state        当前提示词状态
```

### MVP 实现方式

MVP 不必真实使用 `.claude` 文件结构，可先用数据库字段或 JSON 状态模拟。

### 验收标准

- 系统能保存基础画像；
- 下一次对话能复用画像；
- 用户可以要求修改记忆；
- 关键记忆变更需要确认。

## 8.9 Skill / Prompt 动态引导

### 功能说明

根据用户画像切换不同 Prompt 或 Skill 引导策略。

### 核心能力

1. 拆解用户话语；
2. 判断用户画像；
3. 对抗模糊表达；
4. 生成引导问题；
5. 生成 Plan；
6. 根据用户反馈修正路径；
7. 输出文档。

### MVP 约束

- 初期可由单 Agent 模拟多个角色；
- Skill 作为提示词策略，不作为复杂技术系统；
- Multi-Agent 和 MCP 后置。

# 10. AI 行为规则

## 10.1 基础行为原则

AI 不应只回答问题，而应承担引导角色。

基本原则：

1. 先识别用户，再给建议；
2. 先澄清目标，再输出计划；
3. 先判断工具能力，再推荐工具；
4. 先生成可讨论 Plan，再输出最终文档。

## 10.2 追问规则

以下情况必须追问：

1. 用户身份不清；
2. 用户目标不清；
3. 用户工具能力不清；
4. 用户问题过大；
5. 用户想法无法执行；
6. 用户要求跨越过多阶段。

## 10.3 Plan 输出规则

Plan 必须满足：

1. 有用户画像；
2. 有当前问题判断；
3. 有推荐路径；
4. 有执行步骤；
5. 有风险提示；
6. 有下一步选择。

## 10.4 用户画像修正规则

当用户否认系统判断时，系统应：

1. 承认判断可能不准确；
2. 询问修正信息；
3. 更新用户画像；
4. 重新生成或调整 Plan。

## 10.5 禁止行为

AI 不得：

1. 在用户目标不清时直接输出完整长方案；
2. 把普通用户直接推向复杂论文、数据库或专业工具；
3. 生成危险实验步骤；
4. 承诺保证科研成功；
5. 替代专业导师、医生、律师或安全专家；
6. 输出大段用户看不懂的术语堆砌；
7. 把 Markdown 源码当作最终文档展示。

# 14. 风险与应对

## 14.1 愿景过大，MVP 失焦

风险：

> “人人都能做科研”容易导致产品同时服务所有人、所有场景、所有科研流程。

应对：

> MVP 只服务 Z 世代科研新手，先跑通一个完整闭环。

## 14.2 产品退化为普通聊天机器人

风险：

> 如果只是聊天框加 AI 回复，产品缺少差异化。

应对：

必须强化：

1. 用户画像识别；
2. 高频追问；
3. Plan 独立展示；
4. 文档预览。

## 14.3 过早引入复杂技术架构

风险：

> Agent、Skill、MCP、Memory 过早复杂化，会拖慢 MVP。

应对：

MVP 先用单 Agent + Prompt 模板 + 状态存储模拟完整体验。

## 14.4 用户画像过度抽象

风险：

> 清朝人、1930 年以前的人等历史画像有新意，但容易干扰主线。

应对：

历史画像只作为后续验证集，不进入 MVP 主流程。

## 14.5 Plan 质量不稳定

风险：

> 如果 Plan 泛泛而谈，用户不会感知产品价值。

应对：

建立人工审核标准：

1. 是否贴合用户画像；
2. 是否具体；
3. 是否能执行；
4. 是否有风险提示；
5. 是否语言风格匹配用户。

# 16. 最终决策摘要

## 核心结论

**人人都能做科研**的 MVP 正式定位是：

> 面向非专业科研人群的 AI 科研启蒙与路径引导平台。

它的核心不是替专业人士完成科研，而是让普通人通过简单对话进入科研思维，并获得适合自身基础的探索路径。

## 当前评分

**8 / 10**

相比审查版，本版已经完成三项收束：

1. 将长期愿景和 MVP 明确分层；
2. 将 Plan 定义为核心产品形态；
3. 将 Agent / Skill / MCP 降级为后续实现手段。

## 最大风险

**愿景过大导致 MVP 失焦。**

如果一开始就追求全年龄、全画像、全科研流程，产品会迅速变成无法交付的大平台。

## 最可能失败原因

1. 同时服务太多用户；
2. 同时做太多技术模块；
3. 没有先跑通一个用户画像；
4. Plan 输出泛泛；
5. 前端只是聊天框，没有形成 Plan 和文档展示差异化。

## 更稳妥路径

先做一个极小闭环：

> Z 世代科研新手 → 对话输入 → 画像识别 → 高频追问 → Plan 展示 → 文档预览 → 人工审核迭代。

## MVP

MVP 只保留 6 个核心能力：

1. 对话入口；
2. 用户画像识别；
3. 多轮博弈；
4. Plan 生成；
5. Plan 展示；
6. 文档预览。

## 修改优先级

### 必须改

- 继续压缩 P0 功能；
- 画出 Plan 区和文档区原型；
- 制定 Z 世代用户画像测试样例；
- 明确 AI 每轮追问规则。

### 应该改

- 补充用户画像字段；
- 补充人工审核表；
- 补充移动端展示规则；
- 补充 Plan 模板样例。

### 可后置

- 多 Agent；
- MCP；
- 历史时代画像；
- 硬件入口；
- 科研数据库接入。

### 不值得现在做

- 同时支持全年龄段；
- 完整论文科研流程；
- 复杂文件系统；
- 完整 Claude Code 式工程架构。

## 是否建议采用

**建议采用。**

本版可以作为 MVP 正式 PRD，但下一步必须进入更具体的交互原型和 Prompt 工作流设计。

## 信心等级

**中高。**

产品理念有辨识度，Plan + 博弈 + 傻瓜式科研入口具备差异化。成败关键在于：先跑通一个用户画像，而不是直接做“人人都能用”的完整平台。
