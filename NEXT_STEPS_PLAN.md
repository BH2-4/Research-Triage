# Research Triage 后续开发计划

## 1. 当前状态

当前仓库已经完成第一版黑客松 MVP 的基础闭环：

- 首页：明确“先分诊再回答”的产品定位。
- Intake 页：通过中文引导式表单收集任务类型、卡点、基础、截止时间、目标和课题描述。
- 结果页：展示用户画像、当前阶段、风险、最低可行路径和单一服务推荐。
- API：`POST /api/triage/intake` 已可返回规则驱动的分诊结果。
- 基础校验：`typecheck`、`test`、`build` 已通过。

接下来要做的重点，不是把产品做大，而是把“分诊能力”和“商业闭环”做深。

## 2. 总体目标

下一阶段目标分为四条主线：

1. 把当前分诊结果做得更稳定、更像真实客服接待。
2. 把“查看完整项目路线”和“继续免费问 3 个问题”从占位入口升级成可演示功能。
3. 补齐最基本的数据留存、运营分析和安全边界。
4. 为后续部署、展示和继续迭代建立清晰工程结构。

## 3. 阶段一：强化现有分诊闭环

### 目标

让当前 MVP 从“能跑”提升到“更可信、更稳定、更适合演示”。

### 需要完成的内容

- 优化 `triage` 规则：
  - 细化 `userProfile` 的分类规则，让“完全小白型 / 基础薄弱型 / 普通项目型 / 科研能力型 / 焦虑决策型”之间差异更明显。
  - 细化 `taskCategory` 的映射，避免不同问题被过度归到同一类。
  - 调整 `recommendedService` 的推荐逻辑，确保不会一味推高价。
- 优化结果页输出质量：
  - `plainExplanation` 保持 2-4 句、弱术语、强解释。
  - `minimumPath` 的第一步继续保持“24 小时内可执行”。
  - `riskList` 明确体现真实失败点，例如数据、时间、能力、展示和老师预期。
- 增加更多测试样例：
  - 同一课题，不同 `backgroundLevel` 时输出应明显不同。
  - 同一课题，不同 `currentBlocker` 时第一步和推荐服务应变化。
  - 焦虑型、科研型、项目型、小白型都要有对应测试。
- 改善空状态和错误状态：
  - 结果页无数据时给出更明确回流入口。
  - API 失败时给出清晰错误提示，不让页面停在模糊状态。

### 验收标准

- 至少补齐 8-12 个规则测试用例。
- 所有用户类型都能被稳定命中。
- 不出现“解释很长但下一步很虚”的结果。

## 4. 阶段二：实现“完整项目路线”能力

### 目标

把结果页里的“查看完整项目路线”从占位按钮变成真实功能，这是后续最重要的第二步。

### 功能定义

新增一个“项目路线页”或“路线面板”，基于现有 intake 和 triage 结果，生成更长的执行方案，内容至少包括：

- 课题拆解：
  - 研究对象
  - 输入数据
  - 输出结果
  - 最低可交付成果
- 技术路线建议：
  - 更适合先做解释、复现还是 Demo
  - 是否建议先降级为 MVP
  - 是否适合公开数据、模型调用或方法复现
- 阶段计划：
  - 今天
  - 3 天内
  - 1 周内
  - 交付前
- 风险与兜底：
  - 如果数据拿不到怎么办
  - 如果模型做不动怎么办
  - 如果老师要求太高怎么办
- 汇报口径：
  - 怎样向老师解释当前阶段目标
  - 怎样说明为何先做 MVP 或降级方案

### 技术实现建议

- 新增接口：
  - `POST /api/triage/route-plan`
- 输入：
  - intake 原始数据
  - triage 结果
- 输出：
  - `overview`
  - `deliverables`
  - `routeSteps`
  - `fallbackPlan`
  - `teacherTalkingPoints`
- 第一版仍以规则和模板为主，不强依赖联网检索。
- 如果后续接入模型，优先采用“规则骨架 + 模型填充文案”的方式，避免一开始就做长 prompt 黑箱。

### 验收标准

- 路线页必须明显比结果页更深一层，但仍保持结构清晰。
- 至少能支持 3 类典型场景：
  - 小白理解型
  - 本科项目 MVP 型
  - 焦虑交付型

## 5. 阶段三：实现“继续免费问 3 个问题”能力

### 目标

把结果页的第二个入口做成轻量追问机制，而不是完整聊天系统。

### 功能定义

- 用户进入免费追问页后，系统自动带入：
  - intake 信息
  - triage 结果
  - 当前推荐路径
- 用户可以继续问 3 次，每次都应该围绕当前课题推进，而不是重新开始。
- 每次回答都要遵守当前分诊身份和风格：
  - 小白型：人话解释
  - 项目型：任务拆解
  - 科研型：方法比较
  - 焦虑型：边界、兜底和决策支持

### 技术实现建议

- 新增接口：
  - `POST /api/triage/follow-up`
- 新增轻量会话结构：
  - `sessionId`
  - `remainingFreeQuestions`
  - `messages`
- 第一版可以先只把会话放在前端 session storage 或内存里，不急着接数据库。
- 关键限制：
  - 不展开成无限聊天
  - 不偏离当前课题
  - 不绕开安全边界

### 验收标准

- 免费追问必须体现“基于分诊结果继续追问”，而不是普通聊天框。
- 三次问答结束后能自然提示更适合进入哪类服务。

## 6. 阶段四：接入数据留存与运营分析

### 目标

开始记录真实使用数据，为后续产品判断和商业化做准备。

### 最小数据模型

- `triage_requests`
  - `id`
  - `created_at`
  - `task_type`
  - `current_blocker`
  - `background_level`
  - `deadline`
  - `goal_type`
  - `topic_text`
- `triage_results`
  - `request_id`
  - `user_profile`
  - `task_category`
  - `current_stage`
  - `difficulty`
  - `recommended_service`
  - `safety_mode`
- `conversion_events`
  - `request_id`
  - `event_type`
  - `event_value`
  - `created_at`

### 需要记录的行为

- 首页点击“立即诊断课题”
- 表单提交成功
- 结果页点击“查看完整项目路线”
- 结果页点击“继续免费问 3 个问题”
- 是否命中 `safetyMode`
- 用户最常见的 `taskType` 和 `currentBlocker`

### 技术实现建议

- 黑客松后续阶段优先选 SQLite 或 Supabase。
- 数据写入放在 API 层，不放在前端。
- 先保证可查和可统计，不急着引入复杂后台。

### 验收标准

- 能回答最基础的运营问题：
  - 哪类用户最多？
  - 哪类卡点最多？
  - 哪类服务推荐最常出现？
  - 哪个入口点击率最高？

## 7. 阶段五：补齐安全边界与内容审查

### 目标

把“合规辅导模式”从简单关键词判断，升级成更稳的安全策略。

### 需要完成的内容

- 扩展学术不端风险识别：
  - 代写
  - 伪造数据
  - 伪造实验
  - 绕过查重
  - 替做报告或论文
- 增加风险响应策略：
  - 拒绝不合规生成
  - 自动切换到真实交付建议
  - 提供与老师沟通的合理话术
- 为所有输出增加审查检查项：
  - 是否鼓励作弊
  - 是否鼓励伪造结果
  - 是否直接替代学生完成成果

### 验收标准

- 不端输入必须稳定触发 `safetyMode`。
- 返回内容不能出现替代完成、包装伪造或规避审查的建议。

## 8. 阶段六：演示、部署与展示准备

### 目标

确保项目可以稳定上线、稳定演示，并适合对外展示。

### 需要完成的内容

- 部署准备：
  - 补充 `README.md`
  - 写清本地启动、测试和构建命令
  - 准备环境变量说明
- 演示准备：
  - 准备 3 组演示案例
  - 每组案例覆盖不同用户类型
  - 每组案例都能展示“同课题不同路径”
- 页面体验优化：
  - 加载态更自然
  - 移动端更稳定
  - 结果页支持复制与回看
- 若接入线上环境：
  - 优先部署到 Vercel
  - 确保 API 路由正常工作

### 推荐演示脚本

1. 小白型用户：看不懂课题，只想先知道在做什么。
2. 项目型用户：要做课程项目或毕设，需要 MVP 路线。
3. 焦虑型用户：时间很短，怕交付失败，需要兜底方案。

## 9. 推荐开发顺序

建议严格按下面顺序推进：

1. 先强化现有分诊规则和测试。
2. 再实现“完整项目路线”能力。
3. 再实现“继续免费问 3 个问题”能力。
4. 然后补数据留存和事件埋点。
5. 最后做更稳的安全检查和上线准备。

这个顺序的原因是：

- 当前最重要的是证明分诊价值，而不是把系统做成泛聊天产品。
- “完整项目路线”是最接近付费价值的下一步。
- 免费追问适合放在路线能力之后，否则容易退化成普通问答。

## 10. 明确不做的事

为了避免项目失焦，以下内容暂时不进入下一阶段主线：

- 完整文献检索平台
- PDF 解析系统
- 长对话科研助手
- 多 Agent 科研自动化平台
- 真正的支付流程
- 复杂 CRM 后台
- 全自动论文生成

## 11. 最近一次迭代的建议目标

如果只做下一轮最有价值的一个版本，建议目标是：

**“在现有分诊结果基础上，补出完整项目路线页，并让它成为结果页主 CTA 的真实落点。”**

这是最能体现产品价值、最容易继续演示、也最接近后续商业化的一步。
