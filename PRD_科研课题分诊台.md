# PRD：科研课题分诊台 — 面向学生科研项目的 AI 分层客服系统

## 一、核心问题：学生不是缺 AI，而是缺"问题入口"

现在学生面对 AI for Science 的真实状态通常是：

### 1. 信息过载

- AI 能给很多论文、术语、方法、方向。
- 但学生看不懂，也不知道哪些重要。

### 2. 课题陌生

- 拿到项目、比赛题、导师任务、科研方向后，不知道从哪里切。
- 不知道该问背景、方法、实验、代码、论文，还是应用场景。

### 3. 能力差异巨大

- 有些学生能读论文、写代码、做实验。
- 有些学生连关键词、基础概念、研究问题都不清楚。

### 4. 焦虑强

不是单纯要答案，而是想确认：

- 我是不是理解错了？
- 这个课题难不难？
- 我能不能做出来？
- 有没有更简单的版本？
- 我该怎么向老师解释？

**所以这个项目最应该抓住的不是"科研自动化"，而是：科研新手的第一入口 + 分层式科研陪跑客服。**

---

## 二、产品定位：不要叫"科研助手"，要叫"课题理解客服"

如果你把它定义成普通 AI 科研助手，会立刻撞上大量同质化产品。

更清晰的定位应该是：

### 产品一句话

一个面向学生科研项目的 AI 客服网站，先判断学生水平与课题状态，再用不同服务等级帮助学生理解课题、拆解任务、建立安全感，并引导到付费服务。

### 更锋利的表达

学生把课题、论文、比赛想法、导师要求丢进来，系统先判断"你是谁、你卡在哪、你适合做到哪一层"，再给出可执行的科研路径。

这比"我帮你查论文"更好。因为"查论文"只是功能。"我判断你能不能做，并告诉你下一步怎么走"才是价值。

## 三、用户分层：这是你商业化的核心

你的直觉是对的。学生与学生之间差别很大，不能给同一种回答。

建议把用户分为 5 类。

### A 类：完全小白型

**特征**

- 拿到课题但看不懂。
- 不知道关键词。
- 不知道研究目标。
- 不知道怎么问 AI。
- 容易焦虑，容易放弃。

**他们真正需要**

不是论文，不是代码，而是：

- 课题翻译成人话；
- 告诉我这事到底在研究什么；
- 告诉我我能不能做；
- 给我一个最简单版本；
- 告诉我问老师什么。

**对应回答方式**

要低门槛、强解释、少术语。

示例：你现在不需要先读大量论文。你现在最重要的是弄清楚三件事：这个课题解决什么问题、需要哪些基础能力、最低可交付成果是什么。

**商业价值**：高。因为小白用户最依赖"陪跑感"和"安全感"。但他们付费能力可能不一定最高，需要做低价入口。

### B 类：基础薄弱但愿意做型

**特征**

- 能听懂部分解释。
- 愿意做，但缺路线。
- 容易在执行中卡住。
- 需要频繁确认。

**他们真正需要**

- 分步骤任务；
- 每一步的学习材料；
- 每一步的产出模板；
- 遇到问题时有客服式解释。

**对应回答方式**

要给"每日任务 + 检查点"。

示例：你今天只需要完成两件事：第一，确认课题关键词；第二，找到 3 篇能看懂摘要的论文。不要急着写综述。

**商业价值**：最高之一。因为这类人会持续使用，不是一次性提问。适合订阅制、陪跑包、阶段包。

### C 类：普通本科项目型

**特征**

- 有课程设计、毕设、大创、竞赛项目。
- 目标不是顶级科研，而是完成一个像样成果。
- 需要 Demo、汇报、文档、PPT、答辩。

**他们真正需要**

- 可落地项目路线；
- 技术栈选择；
- MVP 设计；
- 文档结构；
- 答辩解释；
- 风险规避。

**对应回答方式**

要偏产品经理 + 技术助教。

示例：这个课题不建议直接做完整科研系统，建议先做一个可演示 MVP：输入数据、调用模型、生成分析报告、前端展示结果。

**商业价值**：很高。这类用户明确有交付需求，愿意为"少走弯路"付费。

### D 类：有能力科研型

**特征**

- 能读论文。
- 有代码能力。
- 对方法、实验、创新点感兴趣。
- 希望快速判断方向价值。

**他们真正需要**

- 论文脉络；
- 方法对比；
- 实验设计；
- 创新点判断；
- 可发表性判断；
- 技术路线审查。

**对应回答方式**

要高密度、高判断、少废话。

示例：这个方向的主要风险不是模型选择，而是数据集和评价指标。如果没有稳定数据来源，建议把创新点收敛到方法复现与场景适配。

**商业价值**：中高。这类用户单价可以更高，但他们不需要太多情绪价值，更看重专业判断。

### E 类：高焦虑决策型

**特征**

- 已经被课题压住。
- 反复问"能不能做""会不会失败""老师会不会不认可"。
- 不是不知道信息，而是不敢决策。

**他们真正需要**

- 课题风险评估；
- 可行性评分；
- 最低交付路径；
- 失败兜底方案；
- 和老师沟通的话术。

**对应回答方式**

要稳定、确定、给边界。

示例：这个项目不是不能做，而是不能按完整科研路线做。你应该改成工程演示型 MVP，保留科研背景，但降低实验深度。

**商业价值**：很高。因为焦虑用户愿意为确定性付费。

## 四、服务分级：不要只按功能分，要按"确定性"分

很多 AI 产品定价失败，是因为它们按功能分级：

- 免费：问答
- 会员：更多次数
- 高级会员：更多模型

这不够锋利。你这个项目应该按用户真正购买的东西分级：他们买的不是回答次数，而是确定性、安全感、交付路径和人工/半人工审查。

### 免费层：诊断入口

**目标**：让用户快速感受到：它懂我现在卡在哪。

**功能**

- 上传/输入课题；
- 选择身份：本科生、研究生、竞赛队伍、大创项目、毕设项目、论文阅读
- 系统输出：
  - 课题人话解释；
  - 难度评分；
  - 当前最该问的 5 个问题；
  - 适合你的路线类型。

**免费层不能给太完整**

免费层要解决"信任建立"，但不能把完整方案全送出去。建议免费层只给：

- 初步判断；
- 三个关键风险；
- 一个最小下一步；
- 推荐服务路径。

### 基础层：课题理解包

**目标**：帮助小白知道自己拿到的东西是什么。

**输出**

- 课题背景解释；
- 核心关键词；
- 研究目标；
- 基础知识补课清单；
- 适合新手的提问清单；
- 3 天入门任务。

**适合用户**：A 类、B 类。

**定价逻辑**：低价，作为转化入口。比如：单次 9.9 / 19.9 / 29.9；或每天 3 次深度解析。

### 进阶层：项目路线包

**目标**：让学生从"知道课题"进入"知道怎么做"。

**输出**

- 技术路线；
- MVP 方案；
- 阶段计划；
- 资料阅读顺序；
- Demo 设计；
- 风险清单；
- 向老师汇报的话术。

**适合用户**：B 类、C 类、E 类。

**定价逻辑**：这是最有商业价值的一层。可以做成：单课题路线生成、7 天项目陪跑、阶段式解锁。

### 高级层：成果交付辅导包

**目标**：围绕具体成果进行陪跑。

**输出**

- 项目文档结构；
- 汇报 PPT 大纲；
- 实验记录模板；
- 代码模块拆解；
- 答辩问答；
- 修改建议；
- 阶段检查。

**适合用户**：C 类、D 类。

**定价逻辑**：按项目或阶段收费。

| 服务 | 价值 |
|------|------|
| 开题理解 | 帮你知道做什么 |
| 路线设计 | 帮你知道怎么做 |
| Demo 设计 | 帮你做出能展示的东西 |
| 答辩准备 | 帮你解释清楚成果 |
| 阶段复盘 | 帮你避免跑偏 |

### 顶层：人工/专家兜底层

**目标**：解决 AI 不够确定的问题。

**适合场景**

- 用户付费意愿高；
- 项目风险高；
- 需要人工审查；
- 需要个性化判断；
- 需要商业级交付。

**定价逻辑**：这里可以高价。因为卖的是：安全感、专业背书、方向纠偏、降低返工。

## 五、定价博弈：你卖的是"焦虑降低程度"

这个项目的定价不能只看 token 成本。

用户付费的真实动机是：

| 用户痛点 | 付费动机 |
|---------|---------|
| 看不懂课题 | 买解释 |
| 不知道怎么做 | 买路径 |
| 怕做不出来 | 买兜底 |
| 怕老师不认可 | 买话术 |
| 怕项目失败 | 买风险控制 |
| 时间不够 | 买压缩路径 |
| 资料太多 | 买筛选 |
| 能力不足 | 买陪跑 |

所以你应该把服务价格锚定在：用户为了减少返工、减少焦虑、减少试错，愿意支付多少钱。而不是：这次 AI 调用了多少 token。

### 定价建议

**1. 低价入口**（适合小白用户）

- 9.9：课题人话解释
- 19.9：课题诊断 + 提问清单
- 29.9：入门路径 + 3 天任务

目的不是赚钱，而是筛选用户。

**2. 中价主力**（适合大多数项目用户）

- 69：完整课题理解包
- 99：项目路线包
- 199：MVP 设计 + 阶段计划

这是主力收入层。

**3. 高价服务**（适合焦虑强、交付压力大的用户）

- 399：完整项目陪跑 7 天
- 699：科研/竞赛项目方案审查
- 999+：人工介入 + 多轮修改 + 答辩准备

这里要谨慎，不要一开始就做重服务。

黑客松阶段只需要证明它有商业潜力。

## 六、情绪价值：不是安慰，而是"结构化安全感"

你提到"承载客户情绪价值"，这个很重要。但情绪价值不能做成鸡汤。真正有效的是：把用户的混乱感变成可控感。

### 系统应该给用户 5 种安全感

**1. 定位安全感**

告诉用户：你现在不是废，而是处在"课题陌生期"。

示例：你现在卡住是正常的，因为这个课题同时包含背景理解、方法选择和交付形式三个问题。我们先不用急着查论文，先把课题拆开。

**2. 难度安全感**

告诉用户：这个项目有难度，但可以降级。

示例：如果按完整科研标准做，难度较高；但如果按本科项目 MVP 做，可以把目标降为"可演示系统 + 合理实验说明"。

**3. 路径安全感**

告诉用户：下一步非常具体，不需要自己乱想。

示例：你现在只需要完成第一步：确认输入数据是什么。没有数据之前，不要急着选模型。

**4. 兜底安全感**

告诉用户：做不成 A，还有 B。

示例：如果真实实验数据拿不到，可以改成公开数据集验证；如果模型训练做不动，可以改成模型调用与结果分析系统。

**5. 解释安全感**

告诉用户：你能向老师解释得过去。

示例：你可以这样向老师表达：当前阶段先完成系统原型，后续再补充实验验证，这样更符合本科项目周期。

## 七、客服路径：锁定定位、需求、安全感、单价

你说的"客服锁定定位，锁定需求，锁定安全感，锁定单价"很关键。可以设计成一个四步转化漏斗。

### 第一步：锁定定位

系统先问：你现在拿到的是哪类任务？

选项：课程项目、毕设、大创、竞赛、导师课题、论文阅读、组会汇报、个人科研探索

然后问：你现在最卡的是哪里？

选项：看不懂题目、不知道查什么、不知道怎么做、不知道能不能做出来、不知道怎么写文档、不知道怎么汇报、老师要求不清楚、已经做了但感觉跑偏

这一步不是普通问卷，而是商业筛选。

### 第二步：锁定需求

系统根据回答判断用户需要哪类服务：

| 用户状态 | 推荐服务 |
|---------|---------|
| 看不懂课题 | 解释 |
| 不知道问什么 | 提问引导 |
| 不知道怎么做 | 路线规划 |
| 不知道能不能做 | 可行性评估 |
| 要交付 | MVP 方案 |
| 要汇报 | PPT/答辩口径 |
| 已经跑偏 | 项目审查 |

这一步要让用户觉得：它知道我真正需要什么。

### 第三步：锁定安全感

给用户一个初步诊断。

格式建议：

```
你现在的问题不是资料不够，而是课题没有被拆开。

当前课题可以分成三块：
1. 背景理解
2. 技术路线
3. 最小可交付成果

你的最低可行路径是：
先做课题理解，再做 MVP，不建议一开始追求完整科研创新。
```

这一段是转化关键。

### 第四步：锁定单价

不要直接说"买会员"。要说：根据你的状态，推荐你选择 X 服务，因为它能解决你当前最关键的问题。

示例：

```
你现在适合的是「项目路线包」，不是「论文精读包」。

原因：
- 你还没有确定项目要做成什么；
- 直接读论文会继续增加信息负担；
- 你更需要先确定 MVP、技术路线和交付物。

推荐服务：项目路线包
预计解决：课题拆解、技术路线、MVP、阶段计划、风险判断。
```

这就是客服式转化。

## 八、黑客松 MVP：不要做大平台，做一个强闭环

黑客松最忌讳做成：AI 科研平台、文献管理系统、论文问答系统、项目管理平台、学术搜索引擎、全能科研助手。

你们应该做一个很窄但很强的 MVP。

### MVP 名称建议

- 课题急诊室
- Research Intake
- 科研客服台
- Topic Triage
- AI 科研分诊台
- 课题分诊助手

其中"分诊"这个词很适合。因为你们不是立刻解决所有问题，而是先判断用户状态，然后分流到不同服务。

### MVP 核心流程

**Step 1：用户输入**

用户输入：课题标题、老师要求、当前基础、截止时间、目标类型、当前卡点。

也可以上传：题目截图、项目说明、论文摘要、比赛需求、导师任务描述。

**Step 2：AI 分诊**

系统输出用户画像：

```
用户类型：基础薄弱但愿意做型
任务类型：本科项目 / AI for Science 入门项目
当前阶段：课题理解期
主要风险：目标过大、方法不清、交付物不明确
推荐路径：先做 MVP，不建议直接追求科研创新
```

**Step 3：生成三层回答**

同一个课题，系统给三种版本：

1. **小白版** — 人话解释；不讲复杂公式；告诉他第一步干什么。
2. **执行版** — 任务拆解；时间计划；工具建议；可交付物。
3. **进阶版** — 论文方向；方法选择；创新点；风险评估。

这可以直接体现"针对性区分"。

**Step 4：服务推荐**

系统给出：

```
你当前最适合的服务：
A. 免费继续问 3 个问题
B. 生成完整项目路线
C. 生成 7 天执行计划
D. 进入人工/专家审查
```

黑客松演示时，这一步就是商业化亮点。

## 九、前端网站怎么设计

你说"用户向前端网站"，这个前端不能只是聊天框。要像一个"科研客服接待台"。

### 首页结构

**第一屏**

标题：你拿到课题后，不知道从哪里开始？

副标题：输入你的课题，我们先判断你处在哪个阶段，再给你适合的科研路径。

按钮：

- 立即诊断课题
- 我不知道怎么描述，让 AI 引导我

### 输入页

不要直接放一个空白聊天框。应该用引导式表单：

```
1. 你的任务类型是什么？
[毕设] [大创] [竞赛] [课程项目] [导师课题] [论文阅读]

2. 你现在最卡在哪里？
[看不懂] [不知道查什么] [不知道怎么做] [怕做不出来] [不知道怎么汇报]

3. 你的基础如何？
[完全小白] [有一点基础] [能读论文] [能写代码] [有实验经验]

4. 你的截止时间？
[3天内] [1周内] [1个月内] [更久]

5. 粘贴你的课题/要求：
[输入框]
```

这样能减少用户不知道怎么问的问题。

### 结果页

结果页建议分为 5 块。

**1. 你的当前状态**

```
你不是缺资料，而是处在"课题理解期"。
当前最重要的是先确定研究对象、输入输出和最低交付物。
```

**2. 课题人话解释**

把复杂课题翻译成人话。

**3. 难度与风险**

```
难度：中等偏高
主要风险：
- 数据来源不明确
- 方法过大
- 交付物不清晰
```

**4. 推荐路径**

```
最低可行路线：
第 1 步：确定研究对象
第 2 步：找到可用数据
第 3 步：做一个最小 Demo
第 4 步：补充解释和展示材料
```

**5. 推荐服务**

```
推荐：项目路线包
原因：你现在最缺的是路径，不是更多资料。
```

## 十、AI 后端逻辑：本质是"分诊 + 路由 + 回答模板"

不要一开始就做复杂 Agent。黑客松 MVP 可以用很简单的架构：

```
用户输入
  ↓
用户画像分类器
  ↓
任务类型分类器
  ↓
风险评估器
  ↓
回答策略路由器
  ↓
生成分层回答
  ↓
推荐服务等级
```

### 1. 用户画像分类器

输入用户基础、任务、表达方式，判断：小白型、基础薄弱型、普通项目型、科研能力型、焦虑决策型。

### 2. 任务类型分类器

判断任务属于：课题理解、文献入门、技术路线、项目 Demo、实验设计、汇报答辩、风险审查。

### 3. 风险评估器

判断：目标是否过大；数据是否缺失；技术是否超出用户能力；时间是否不足；是否需要降级 MVP；是否存在学术诚信风险。

**这里要加一个安全边界**：系统不应该帮助学生伪造科研成果、伪造实验数据、代写论文或规避学术审查。它应该帮助学生理解课题、拆解任务、规划学习、准备真实交付。这会让项目在黑客松展示时更稳。

### 4. 回答策略路由器

不同用户给不同回答。

| 用户 | 回答风格 |
|------|---------|
| 小白 | 人话解释 + 少术语 |
| 基础薄弱 | 步骤清单 + 学习路径 |
| 普通项目 | MVP + 技术路线 |
| 科研能力强 | 方法对比 + 风险评估 |
| 焦虑用户 | 判断 + 兜底方案 + 汇报话术 |

## 十一、AI token 成本如何下降

你之前提到 token 成本问题。这个项目如果按"长对话客服"做，成本会炸。所以需要从一开始就控制。

### 1. 先分类，后深度回答

不要用户一输入就生成长答案。先用低成本模型做：用户分类、任务分类、风险判断、推荐回答模板。只有用户点击"生成详细路线"时，才调用高成本模型。

### 2. 使用模板化回答骨架

很多内容不需要每次完全生成。例如：小白解释模板、难度评分模板、MVP 路线模板、服务推荐模板、风险清单模板。AI 只填关键变量。这样可以减少 token，同时保证稳定。

### 3. 分层生成

免费层短回答，付费层长回答。

| 层级 | 输出长度 |
|------|---------|
| 免费诊断 | 300-600 字 |
| 基础包 | 1000-1500 字 |
| 路线包 | 2000-3000 字 |
| 高级包 | 多轮生成 |

### 4. 缓存相似课题

很多学生问题高度相似。例如：AI + 药物发现、AI + 材料筛选、AI + 蛋白质预测、AI + 图像识别、AI + 时间序列预测、AI + 文献综述。可以把常见课题做成模板库。用户输入后先匹配模板，再局部生成。

### 5. 不要默认联网检索

黑客松阶段不一定要做真实文献检索。你们可以先做：课题理解、路线规划、用户分诊、服务推荐。真实文献检索可以作为进阶功能，而不是 MVP 核心。

## 十三、质量如何保证少返工

你们的核心交付不是"答案多"，而是"答案稳定"。

建议设置 4 个检查器。

### 1. 定位检查

检查回答是否真的匹配用户类型。例如：小白用户不能给太多论文术语；科研型用户不能给太浅解释；焦虑型用户不能只堆资料。

### 2. 路径检查

检查是否给出可执行下一步。

错误回答：建议你阅读相关论文，深入理解该领域。

正确回答：今天先找出 3 个关键词：研究对象、输入数据、输出结果。然后用这 3 个关键词查 3 篇综述。

### 3. 风险检查

检查是否指出失败点。尤其是：数据拿不到；目标太大；时间不足；用户能力不匹配；Demo 无法展示；结果无法解释。

### 4. 商业推荐检查

检查服务推荐是否合理。不能所有用户都推荐最高价。否则用户会觉得像骗钱。应该是：小白 → 低价理解包；项目型 → 路线包；高焦虑 → 陪跑包；高能力 → 审查包。

## 十四、最大优点

这个项目最大的优点是：你抓住了 AI for Science 里一个更底层的问题：学生不是不会用 AI，而是不知道如何把科研任务转化成自己能理解、能执行、能交付的路径。

它不是在和论文搜索、ChatPDF、文献总结工具正面竞争。它做的是更前置的入口：

```
课题输入
  ↓
用户分诊
  ↓
能力匹配
  ↓
路径生成
  ↓
服务转化
```

这是商业价值更强的位置。因为入口型产品更容易承接用户焦虑，也更容易做服务分级。

## 十五、最大风险

最大风险是：产品容易滑向"泛 AI 问答 + 包装成科研客服"，没有形成真正的分诊能力。

如果最后只是：用户输入课题 → AI 生成一大段解释 → 推荐购买套餐，那就很普通。用户不会觉得这是一个新产品，只会觉得是套壳 ChatGPT。

你们必须证明：

1. 系统能识别不同用户；
2. 同一个课题能给不同层级回答；
3. 系统能判断用户真正缺什么；
4. 推荐服务不是硬推，而是基于用户状态；
5. 输出能降低焦虑并推动行动。

## 十六、最可能失败原因

最可能失败的原因有 5 个。

### 1. 做得太大

想同时做：文献检索、科研规划、项目管理、论文生成、代码生成、答辩辅导、商业客服。黑客松时间不够，会做散。

### 2. 用户分类太虚

如果分类只是标签，没有影响回答和定价，就没有意义。分类必须改变：回答深度、解释方式、推荐路径、服务价格、转化话术。

### 3. 商业化太生硬

不能直接说：购买高级套餐获得更多回答。

要说：你现在最缺的是项目路线，不是继续查资料，所以推荐项目路线包。

### 4. 情绪价值做成空话

不能只说：不要焦虑，你可以的。

要说：你现在卡住是因为缺少研究对象、数据来源和最低交付物，我们先处理这三件事。

### 5. 没有安全边界

如果产品看起来像帮助学生偷懒、代写、伪造科研成果，会在比赛展示里扣分。

要强调：我们不是替学生完成科研，而是帮助学生理解课题、拆解任务、制定真实可执行路径。

---

## 十七、更稳妥、更简单、成功率更高的路径

最稳妥的版本不是"AI 科研平台"，而是：AI 科研课题分诊台。

只做 3 件事：

### 1. 识别用户

- 你是谁？
- 你基础如何？
- 你现在卡在哪？

### 2. 诊断课题

- 这个课题在研究什么？
- 难点在哪里？
- 最低可行成果是什么？

### 3. 推荐路径

- 小白该怎么入门？
- 普通学生怎么做 MVP？
- 有能力的人怎么深入？
- 焦虑用户怎么降低失败风险？

这条路径最适合黑客松。因为它：前端好展示；AI 能力容易体现；商业闭环清晰；不依赖复杂科研数据库；可快速做 Demo；评委容易理解。

## 十八、MVP 版本

**MVP 标题**

科研课题分诊台：面向学生的 AI for Science 客服入口

**MVP 用户流程**

```
用户输入课题
  ↓
选择身份和卡点
  ↓
系统判断用户类型
  ↓
输出课题诊断
  ↓
生成分层回答
  ↓
推荐服务等级
```

**MVP 必须有的 5 个页面**

1. **首页** — 突出一句话：拿到课题不知道从哪问？先让 AI 帮你分诊。
2. **输入页** — 引导式表单，不是空白聊天框。
3. **诊断页** — 输出：用户类型、任务阶段、主要卡点、难度评分、当前最该做什么。
4. **分层回答页** — 展示同一个课题对不同用户的回答差异：小白版、执行版、进阶版。这是黑客松演示的亮点。
5. **服务推荐页** — 展示：免费继续问、课题理解包、项目路线包、陪跑/审查包。

## 十九、修改优先级

### 必须改

1. **不要把项目定义成"AI for Science 信息获取工具"** — 太泛。应改成"科研课题分诊客服"。
2. **必须明确用户分类会影响回答** — 否则分类只是装饰。
3. **必须做服务分级** — 免费诊断、课题理解、项目路线、陪跑审查。
4. **必须有安全边界** — 不代写、不伪造数据、不承诺科研结果、只做理解、规划、辅导和审查。
5. **必须突出商业闭环** — 诊断结果自然导向服务推荐。

### 应该改

1. **弱化"科研信息繁多"这个表述** — 这只是表层痛点。更核心的是"不会提问、不会判断、不会执行"。
2. **强化"客服"而不是"聊天"** — 客服意味着分诊、引导、转化、安抚。聊天只是交互方式。
3. **把情绪价值结构化** — 不做安慰。做定位、风险、兜底、下一步。
4. **把定价逻辑从 token 成本转成确定性价值** — 用户不是为 token 付费。用户为减少焦虑和返工付费。

### 可后置

1. 文献自动检索。
2. PDF 解析。
3. 多 Agent 科研流程。
4. 项目管理看板。
5. 人工专家后台。
6. 长期陪跑系统。
7. 校园渠道分销。

这些都重要，但不适合黑客松第一版。

### 不值得一开始做

1. 完整科研平台。
2. 高级论文阅读器。
3. 大规模知识库。
4. 复杂积分会员系统。
5. 真正的人工客服排班。
6. 过度复杂的 CRM。
7. 全自动科研成果生成。

这些会拖垮 MVP。

## 末尾决策收束

### 核心结论

这个黑客松项目值得做，但不要做成泛 AI for Science 平台。最稳妥的定位是：面向学生科研项目的"AI 课题分诊客服"。

### 评分

8/10。概念有商业潜力，也适合黑客松展示；主要扣分在于当前表述过散，必须收束成"分诊—路径—服务推荐"的闭环。

### 最大风险

做成普通 AI 聊天网站，缺少真正的用户分类、服务路由和商业转化逻辑。

### 最可能失败原因

试图同时做科研助手、文献检索、客服系统、项目管理和商业平台，导致 MVP 过重，演示不聚焦。

### 更稳妥路径

先做：课题输入 → 用户分诊 → 课题诊断 → 分层回答 → 服务推荐。不要一开始做复杂文献检索和完整科研流程。

### MVP

科研课题分诊台。

核心功能：

1. 用户输入课题和当前卡点；
2. 系统判断用户类型；
3. 输出课题人话解释、难度、风险；
4. 给小白版、执行版、进阶版三种回答；
5. 推荐对应服务层级。

### 修改优先级

| 优先级 | 内容 |
|--------|------|
| 必须改 | 收束定位为"科研课题分诊客服" |
| 必须改 | 用户分类必须影响回答和推荐服务 |
| 必须改 | 建立免费层、路线层、陪跑层的服务分级 |
| 应该改 | 情绪价值改成结构化安全感 |
| 应该改 | 定价逻辑从 token 成本转为确定性价值 |
| 可后置 | 文献检索、PDF 解析、人工专家后台 |
| 不值得做 | 一开始做完整 AI for Science 平台 |

### 是否建议采用

建议采用。但只建议采用收束后的版本：AI 科研课题分诊台，而不是 AI 科研万能助手。

### 信心等级

中高。原因是学生科研入门痛点真实，商业化路径也成立；但成败高度依赖产品是否能把"客服分诊感"做出来，而不是只生成一大段 AI 回答。
